# Parquet 포맷

이 앱이 읽기 위해 존재하는 디스크 파일 포맷을 설명합니다.

---

## 1. Parquet이란

[Apache Parquet](https://parquet.apache.org/)은 분석용 데이터를 위한 개방형 **컬럼나** 저장
포맷입니다. CSV나 행 지향 데이터베이스 파일과 달리, Parquet은 한 열의 모든 값을 함께 저장하고,
각 열을 독립적으로 압축하며, 파일 구조와 통계를 기술하는 풍부한 **메타데이터 footer**를 내장합니다.

데이터 레이크와 분석 파이프라인의 사실상 표준 포맷인데, **자기 기술적**(스키마가 데이터와 함께
이동)이고, **압축적**(열별 압축)이며, **선택적으로 읽을 수 있기**(전체 파일을 스캔하지 않고
필요한 열·행 범위만 가져옴) 때문입니다.

## 2. 물리적 구조

```
┌──────────────────────────────────────────────────────────┐
│ "PAR1"  (4바이트 매직 헤더)                                 │
├──────────────────────────────────────────────────────────┤
│ Row Group 0                                                │
│   ├─ Column Chunk: col_a   → [Page][Page][Page] ...        │
│   ├─ Column Chunk: col_b   → [Page][Page] ...              │
│   └─ Column Chunk: col_c   → [Page] ...                    │
├──────────────────────────────────────────────────────────┤
│ Row Group 1                                                │
│   └─ ... 같은 열들, 다음 가로 행 슬라이스                    │
├──────────────────────────────────────────────────────────┤
│ ...                                                        │
├──────────────────────────────────────────────────────────┤
│ Footer  (파일 메타데이터, Thrift 인코딩)                     │
│   ├─ schema (열 이름 + 타입)                                │
│   ├─ row group별: 행 수, 바이트 크기, 압축,                  │
│   │              열별 통계(min/max/…)                        │
│   └─ footer 길이                                            │
├──────────────────────────────────────────────────────────┤
│ "PAR1"  (4바이트 매직 트레일러)                              │
└──────────────────────────────────────────────────────────┘
```

거친 단위에서 세밀한 단위로의 계층:

| 단계 | 담는 것 |
|------|---------|
| **File** | 하나의 Parquet 파일이 담는 전체 데이터셋. 상·하단에 매직 바이트 `PAR1`. |
| **Row Group** | 행의 가로 슬라이스(예: ~100만 행). 각 row group은 해당 행들의 모든 열을 담음. 병렬성과 pushdown의 단위. |
| **Column Chunk** | 한 row group 안에서 한 열의 모든 값 — 연속 저장되고 압축됨. |
| **Page** | column chunk 내부의 최소 단위: 인코딩+압축된 값과 자체 헤더를 가짐. |
| **Footer** | 후행 메타데이터: 스키마, row group 목록, 열별 통계. **마지막**에 읽지만 **가장 먼저** 필요함. |

### 인코딩과 압축

column chunk 안에서 값은 먼저 **인코딩**되고(딕셔너리 인코딩, 런 렝스 인코딩, 비트 패킹 — 한
열의 이웃 값들이 비슷하기에 효과적), 그 다음 **SNAPPY**, **ZSTD**, **GZIP** 같은 코덱으로
**압축**되거나 **UNCOMPRESSED**로 남습니다. 코덱은 footer에 column chunk별로 기록됩니다 — 이 앱은
Row Groups 탭에서 row group별로 이를 표시합니다.

## 3. 컬럼나가 빠르고 선택적인 읽기를 가능케 하는 이유

구조에서 두 가지 최적화가 자연히 나오며, 이 앱이 쿼리를 실행할 때 DataFusion이 둘 다 사용합니다:

- **Projection pushdown** — 쿼리가 `col_a`만 선택하면, 리더는 `col_a`의 column chunk만 가져오고
  나머지 바이트는 통째로 건너뜁니다.
- **Predicate pushdown / row group 스킵** — 각 row group은 footer에 열별 **min/max 통계**를
  기록합니다. `WHERE col_a > 1000`이라면, `col_a` 최댓값이 1000 이하인 row group은 값 하나도
  디코딩하지 않고 건너뛸 수 있습니다.

이 앱의 100행 스트림 정지(
[DataFusion & Arrow §3](./02-datafusion-and-arrow.md#3-스트리밍-실행과-100행-캡) 참고)와 결합하면,
멀티 GB 파일을 열어 미리 보는 작업이 저렴하게 유지됩니다: 엔진은 footer를 읽은 뒤, 앞쪽 100행을
만드는 데 필요한 열/row group 바이트만 읽습니다.

## 4. Footer: 마지막에 있지만 가장 먼저 읽는다

유용한 일을 하려면 리더는 먼저 **footer**를 파싱해야 합니다. 스키마와 row group 레이아웃이 거기
있기 때문입니다. 표준 읽기 순서:

1. 파일 끝으로 seek해 후행 매직 + 4바이트 footer 길이를 읽음.
2. footer 바이트(Thrift 인코딩 메타데이터)를 읽음 — **아직 열 데이터는 없음**.
3. footer로부터 스키마와, 가져올 row group / column chunk를 알아냄.
4. 필요한 page만 가져와 디코딩.

그래서 메타데이터 작업은 파일 크기와 무관하게 거의 즉시 끝납니다 — 데이터가 아니라 footer만
만지기 때문입니다.

## 5. 이 앱이 Parquet을 읽는 방법

### 스키마 (파일 열기 시)

파일이 등록될 때(`src-tauri/src/engine/context.rs::register_source`), DataFusion의
`ListingTableConfig::infer`가 footer를 읽어 Arrow 스키마를 추론합니다. 그 스키마가 `data` 테이블의
열 목록이 되고, Schema 탭을 위해 `SchemaField[]`(이름, Arrow 타입, nullable)로 프론트엔드에
전달됩니다.

### Footer 메타데이터 (파일 열기 시)

등록 직후, 엔진은 파일 통계를 채우기 위해 footer를 명시적으로 읽습니다 — **원시 로컬 파일 핸들을
열지 않으므로**, 동일 경로가 원격 객체에도 동작합니다:

```rust
// context.rs (요약)
let object_meta = object_store.head(&path).await?;          // stat: 위치 + 크기
let mut reader = ParquetObjectReader::new(object_store, object_meta.location)
    .with_file_size(object_meta.size);                      // 크기를 주면 footer로 바로 seek 가능
let parquet_meta = reader.get_metadata(None).await?;        // footer만 읽음 — 행 데이터는 절대 안 읽음
```

이어서 `build_file_metadata()`가 `parquet_meta.row_groups()`를 순회하며 다음을 만듭니다:

```rust
FileMetadata {
    total_rows,                 // 모든 row group에 걸쳐 합산
    row_groups: [ RowGroupInfo { num_rows, total_byte_size, compression }, ... ],
}
```

이것이 **Row Groups 탭**에 표시되며, `total_rows`는 파일의 실제 행 수입니다 — 이는 *쿼리 결과*의
행 수와는 별개임에 유의하세요. 쿼리 결과 행 수는 100행 캡으로 제한됩니다.

모든 Parquet 타입은 직접 `parquet` 크레이트 의존성이 아니라 `datafusion::parquet::*` re-export를
통해 접근합니다 —
[DataFusion & Arrow §6](./02-datafusion-and-arrow.md#6-단일-버전-의존성-규율) 참고.

## 6. Parquet vs Arrow — 둘의 관계

둘은 **같은 아이디어(컬럼나)를 두 곳에서** 구현한 것입니다:

| | Parquet | Arrow |
|--|---------|-------|
| 사는 곳 | **디스크** | **메모리** |
| 최적화 대상 | 저장 크기, 선택적 읽기 | CPU 처리, 제로카피 접근 |
| 압축 | 있음 (SNAPPY/ZSTD/…) | 없음 (원시 타입 버퍼) |
| 이 앱에서 | 소스 파일 | 쿼리가 만든 `RecordBatch`, 그리고 IPC 전송 포맷 |

Parquet 읽기는 본질적으로: column chunk/page 디코딩 → Arrow `RecordBatch` 복원 → DataFusion에
전달입니다. 컬럼나 형태가 파일에서 쿼리 엔진을 거쳐 IPC로 보내는 바이트까지 곧장 이어지므로,
중간에 데이터를 재구성할 필요가 없습니다.

## 7. Parquet vs CSV/TSV — 행 기반 포맷과의 차이

CSV/TSV는 **행 기반(row-oriented)** 텍스트 포맷입니다. 한 줄이 한 레코드이고, 필드는 구분자(`,`
또는 탭)로 나뉩니다. 모든 값이 사람이 읽을 수 있는 텍스트이며, 타입·스키마·통계 같은 메타데이터가
없습니다. Parquet은 이와 정반대 설계입니다.

### 저장 방식의 근본 차이

같은 표를 디스크에 놓는 방식이 다릅니다:

```
원본 표
  id | name  | age
  ---+-------+----
   1 | Kim   | 30
   2 | Lee   | 25
   3 | Park  | 41

CSV (행 기반): 한 행의 모든 열을 연달아 저장
  1,Kim,30⏎2,Lee,25⏎3,Park,41

Parquet (열 기반): 한 열의 모든 값을 연달아 저장
  [id]   1,2,3
  [name] Kim,Lee,Park
  [age]  30,25,41
```

이 한 가지 차이에서 아래의 모든 장단점이 파생됩니다. 예를 들어 `SELECT avg(age)`는 Parquet에서는
`age` 열 블록만 읽으면 되지만, CSV에서는 모든 행의 모든 바이트를 읽고 파싱해 `age` 필드만 골라내야
합니다.

### 항목별 비교

| 항목 | CSV / TSV (행 기반) | Parquet (열 기반) |
|------|---------------------|-------------------|
| 데이터 배치 | 행 단위 (레코드가 연속) | 열 단위 (같은 열 값이 연속) |
| 스키마 | 없음 (있어도 헤더 1줄, 타입 정보 없음) | 내장 (열 이름 + 타입 + nullable) |
| 타입 | 전부 텍스트 → 읽을 때 추론/파싱 필요 | 명시적 타입 (Int64, Double, Timestamp 등) |
| 압축 | 파일 전체를 통째로 gzip 등 (열 무관) | 열별 독립 압축 + 인코딩 (딕셔너리/RLE) |
| 일부 열만 읽기 | 불가 — 전체를 스캔·파싱 | 가능 — 해당 column chunk만 읽음 (projection pushdown) |
| 행 필터링 | 전체 스캔 | row group min/max 통계로 블록 스킵 (predicate pushdown) |
| 파일 크기 | 큼 (텍스트 + 무압축 기본) | 작음 (보통 CSV의 1/3 ~ 1/10) |
| 사람이 직접 읽기 | 가능 (텍스트 에디터로 열림) | 불가 (바이너리 — 전용 도구 필요) |
| 추가(append)·스트리밍 작성 | 쉬움 (한 줄씩 이어 붙임) | 어려움 (footer를 마지막에 한 번 써야 함) |
| 부분 수정 | 텍스트라 비교적 쉬움 | 사실상 불가 (불변 파일로 재작성) |
| 도구·호환성 | 어디서나 열림 (Excel, 메모장 등) | 데이터 도구 생태계 필요 (Spark, DuckDB, 이 앱 등) |

### Parquet의 장점

- **분석 쿼리에 빠름** — 필요한 열·행 그룹만 읽으므로 I/O가 대폭 줄어듭니다. 이 앱이 GB급 파일을
  즉시 미리 볼 수 있는 이유입니다(§3).
- **저장 효율** — 열별로 비슷한 값이 모여 있어 인코딩·압축 효율이 높습니다. CSV보다 훨씬 작습니다.
- **타입 안전 + 자기 기술적** — 스키마와 타입이 파일 안에 있어, 별도 정의 없이도 정확히 해석됩니다.
  CSV처럼 "이 열이 숫자인가 문자열인가"를 추측할 필요가 없습니다.
- **메타데이터 즉시 조회** — footer만 읽어 행 수·통계를 얻습니다(§4).

### Parquet의 단점 (= CSV/TSV가 나은 경우)

- **사람이 직접 못 읽음** — 바이너리라서 텍스트 에디터로 열리지 않습니다. 빠르게 눈으로 확인하거나
  손으로 고치기에는 CSV가 편합니다 (이 앱 같은 뷰어가 필요한 이유이기도 합니다).
- **작성·추가가 무겁다** — footer를 마지막에 한 번에 써야 하므로, 로그처럼 한 줄씩 계속 덧붙이는
  스트리밍 쓰기에는 CSV가 더 단순합니다.
- **부분 수정 불가** — 한 값만 바꾸려 해도 사실상 파일을 다시 써야 합니다.
- **작은 데이터에선 이점이 작다** — 수십~수백 행 규모에서는 압축·pushdown 이득이 미미하고, 오히려
  footer/메타데이터 오버헤드와 도구 의존성이 부담이 될 수 있습니다.

### 정리: 언제 무엇을

- **행 단위로 계속 쓰거나, 사람이 바로 열어 보거나, 어디서나 호환돼야 한다** → CSV/TSV
- **한 번 쓰고 여러 번 분석 쿼리하거나, 데이터가 크거나, 타입·스키마를 보존해야 한다** → Parquet

이 앱은 후자의 시나리오(쓰여진 Parquet을 SQL로 탐색)에 정확히 맞춰져 있습니다.
